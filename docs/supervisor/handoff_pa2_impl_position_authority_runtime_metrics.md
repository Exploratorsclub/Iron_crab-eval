# Handoff PA-2: PositionAuthority Runtime Metrics (Read-only)

## Dependency

PA-1 is merged on `architecture-rebuild`:

- `src/position_authority/`
- `PositionAuthority` pure reducer
- `PositionEvent`
- `PositionState`
- no runtime wiring yet

This PA-2 scope must build on PA-1.

## Task-Beschreibung

Wire `PositionAuthority` into runtime **read-only** so we can observe whether it agrees with current execution state during real bot runs.

Goal:

- Create/passively maintain a PositionAuthority instance in `execution-engine`.
- Feed it from events that execution-engine already sees:
  - confirmed `ExecutionResult` generated by execution-engine
  - `WalletBalanceSnapshot` events consumed from JetStream by execution-engine
- Expose metrics:
  - `position_authority_open_positions`
  - `position_authority_reconcile_needed_positions`
  - `position_authority_drift_lockmanager` = authority open count minus LockManager open count (signed if metric style supports it, or two gauges if easier)
- Do not use PositionAuthority for risk checks, preflight, reservations, dashboard `open_positions`, Momentum, or routing yet.

This scope is visibility only. It must not change trading behavior.

## Relevante Invarianten (Volltext)

## Expected Implementation Direction

### A. ExecutionContext field

Add a field to `ExecutionContext`, e.g.:

```rust
position_authority: Arc<ParkingMutex<PositionAuthority>>
```

or equivalent using existing lock style.

### B. Feed confirmed ExecutionResult

After a confirmed `ExecutionResult` is built/enriched and before/after it is written/published, map it with:

```rust
PositionEvent::try_from_execution_result(&exec)
```

and apply it to the authority.

Important:

- Only confirmed results should affect authority.
- BUY uses fill_out.
- SELL uses fill_in.
- Preserve Token-2022 metadata.

### C. Feed WalletBalanceSnapshot

In the existing execution-engine WalletBalanceSnapshot consumer:

- After parsing `MarketEventKind::WalletBalanceSnapshot`, map with `PositionEvent::try_from_market_event_kind(&event.kind)`.
- Apply it to authority.
- Continue existing LockManager behavior unchanged.

Important:

- SOL/WSOL should not become tradeable authority positions. If PA-1 currently maps all wallet snapshots, add a guard so SOL/WSOL snapshots are ignored for open trade positions, or handle this before applying.

### D. Metrics

Expose:

- `position_authority_open_positions`
- `position_authority_reconcile_needed_positions`
- `position_authority_drift_lockmanager`

If signed gauges are awkward with current metrics helpers, expose:

- `position_authority_open_positions`
- `position_authority_lockmanager_open_positions`
- `position_authority_drift_abs`
- `position_authority_drift_direction` (`1` authority higher, `0` equal, `-1` if signed supported; otherwise omit)

Keep existing `open_positions` unchanged.

### E. Tests

Add tests proving runtime helper behavior where possible:

1. Confirmed BUY ExecutionResult applied to authority increments authority count.
2. Confirmed SELL ExecutionResult applied to authority decrements/closes.
3. WalletBalanceSnapshot non-zero recovers authority position.
4. WalletBalanceSnapshot zero closes/removes authority position.
5. SOL/WSOL snapshots are ignored for authority open position count.
6. Drift helper computes expected values versus a supplied LockManager count.

Use pure helper functions if testing full `ExecutionContext` is too heavy.
